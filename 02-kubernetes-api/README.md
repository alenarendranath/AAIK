# 02 — Kubernetes API & Object Model

## How the Kubernetes API Works

Every action in Kubernetes goes through the **API Server**. It exposes a RESTful API:

```
GET    /api/v1/namespaces/default/pods          → list pods
POST   /api/v1/namespaces/default/pods          → create pod
GET    /api/v1/namespaces/default/pods/aaik     → get pod
PUT    /api/v1/namespaces/default/pods/aaik     → replace pod
PATCH  /api/v1/namespaces/default/pods/aaik     → update pod
DELETE /api/v1/namespaces/default/pods/aaik     → delete pod
```

## Object Manifest Structure

Every Kubernetes object has these four top-level fields:

```yaml
apiVersion: v1          # API group + version (e.g. v1, apps/v1, batch/v1)
kind: Pod               # Resource type
metadata:               # Identity information
  name: aaik-pod
  namespace: default
  labels:
    app: aaik
  annotations:
    description: "AAIK demo pod"
spec:                   # DESIRED state (you define this)
  containers:
  - name: aaik
    image: nginx:alpine
status:                 # ACTUAL state (Kubernetes fills this in)
  phase: Running
```

## API Groups

| Group | Example Resources | apiVersion |
|-------|-------------------|------------|
| Core  | Pod, Service, ConfigMap, Secret, Node | `v1` |
| apps  | Deployment, ReplicaSet, StatefulSet, DaemonSet | `apps/v1` |
| batch | Job, CronJob | `batch/v1` |
| networking | Ingress, NetworkPolicy | `networking.k8s.io/v1` |
| storage | StorageClass, PersistentVolume | `storage.k8s.io/v1` |
| gateway | Gateway, HTTPRoute | `gateway.networking.k8s.io/v1` |

## Understanding Objects

### metadata fields
| Field | Description |
|-------|-------------|
| `name` | Unique within namespace |
| `namespace` | Logical grouping (default: `default`) |
| `labels` | Key/value pairs for selection and organization |
| `annotations` | Non-identifying metadata (longer strings, URLs, etc.) |
| `uid` | Unique ID assigned by Kubernetes |
| `resourceVersion` | Version for optimistic concurrency |
| `ownerReferences` | Links object to its owner (e.g., Pod → ReplicaSet) |

### spec vs status
- **spec** = what YOU want (desired state)
- **status** = what Kubernetes has done (current state)
- Kubernetes constantly reconciles actual → desired

## Node Object Example

```yaml
apiVersion: v1
kind: Node
metadata:
  name: kind-worker
  labels:
    kubernetes.io/arch: amd64
    kubernetes.io/os: linux
    node-role.kubernetes.io/worker: ""
status:
  conditions:
  - type: Ready
    status: "True"
  allocatable:
    cpu: "4"
    memory: 8Gi
  capacity:
    cpu: "4"
    memory: 8Gi
```

## Event Objects

Events are automatically generated by Kubernetes to record what happened:

```yaml
apiVersion: v1
kind: Event
metadata:
  name: aaik-pod.scheduled
reason: Scheduled
message: "Successfully assigned default/aaik-pod to kind-worker"
type: Normal   # Normal or Warning
involvedObject:
  kind: Pod
  name: aaik-pod
```
